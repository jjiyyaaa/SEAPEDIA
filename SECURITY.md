# SEAPEDIA Security Implementation Notes

This document describes how the SEAPEDIA platform implements robust, production-grade security measures across both frontend and backend systems.

## 1. SQL Injection Prevention
- **ORMs & Parameterization**: The backend uses **Prisma ORM** with an SQLite database. Prisma automatically parameterizes all queries generated by its query builder (e.g., `prisma.user.findUnique`, `prisma.store.create`).
- **No Raw SQL**: Raw SQL queries (`$queryRaw` or `$executeRaw`) are avoided entirely in the codebase. This guarantees that user inputs are never concatenated directly into SQL statements, preventing SQL injection vulnerabilities.

## 2. Cross-Site Scripting (XSS) Protection
- **Input Sanitization**: All text-based inputs (including reviews comment, buyer address, store name, product name, and product description) are sanitized in the controllers using a utility escaper before database insertion.
- **Utility Implementation**: The sanitizer ([security.js](file:///c:/Users/ghazi/.gemini/antigravity-ide/scratch/seapedia/backend/src/utils/security.js)) converts HTML special characters to safe entities:
  - `&` -> `&amp;`
  - `<` -> `&lt;`
  - `>` -> `&gt;`
  - `"` -> `&quot;`
  - `'` -> `&#x27;`
  - `/` -> `&#x2F;`
- **Output Safety**: The frontend renders escaped properties safely (using `.textContent` or escaping helpers) so script tags or iframe injections are neutralized and rendered as plain text.

## 3. Input Validation & Numeric Boundary Enforcement
- **Strict Parsing**: Fields like product prices, stock levels, wallet top-ups, and cart quantities are parsed using `parseInt(..., 10)`.
- **Boundary Checks**: The backend enforces `value >= 1` logic, rejecting 0, negative integers, or non-numeric values with a `400 Bad Request` status.
- **Cart Invariant**: Single-Store rule ensures that buyers cannot add items from different stores in a single checkout session.

## 4. Session Behavior & Tokens
- **JSON Web Tokens (JWT)**: Sessions are signed using a secure JWT key (`JWT_SECRET`).
- **Expiration Policy**:
  - Temporary role selection tokens expire in **10 minutes**.
  - Finalized session tokens expire in **1 day**.

## 5. Role-Based Access Control (RBAC)
- **Server-Side Enforcement**: Access control is secured directly on the backend routes via Express middleware, not just hidden on the frontend UI:
  - `verifySeller`: Restricts access to store/product CRUD.
  - `verifyBuyer`: Restricts wallet, cart, and checkout paths.
  - `verifyDriver`: Restricts delivery jobs and couriers dashboard access.
  - `verifyAdmin`: Restricts time simulation and discount code generations.
